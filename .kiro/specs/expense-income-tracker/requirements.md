# เอกสารข้อกำหนดความต้องการ (Requirements Document)

## บทนำ

ระบบเว็บแอปพลิเคชันสำหรับบันทึกรายรับรายจ่าย และวิเคราะห์ข้อมูลทางการเงินจากหลายมุมมอง เพื่อช่วยให้ผู้ใช้สามารถติดตามสถานะทางการเงินของตนเอง วางแผนการใช้จ่าย และเข้าใจพฤติกรรมทางการเงินผ่านการแสดงผลข้อมูลในรูปแบบกราฟและรายงานต่างๆ

## อภิธานศัพท์ (Glossary)

- **System**: ระบบเว็บแอปพลิเคชันบันทึกรายรับรายจ่าย
- **User**: ผู้ใช้งานระบบที่ลงทะเบียนแล้ว
- **Transaction**: รายการทางการเงิน ซึ่งเป็นได้ทั้งรายรับ (income) หรือรายจ่าย (expense)
- **Category**: หมวดหมู่ของรายการทางการเงิน เช่น อาหาร, ค่าเดินทาง, เงินเดือน
- **Dashboard**: หน้าแสดงผลข้อมูลสรุปภาพรวมทางการเงิน
- **Analytics_Module**: โมดูลวิเคราะห์ข้อมูลที่แสดงผลในรูปแบบกราฟและรายงาน
- **Parser**: ตัวแปลงข้อมูลที่นำเข้าจากไฟล์ภายนอก
- **Pretty_Printer**: ตัวจัดรูปแบบข้อมูลสำหรับส่งออกเป็นไฟล์

## ข้อกำหนดความต้องการ (Requirements)

### ข้อกำหนดที่ 1: การลงทะเบียนและเข้าสู่ระบบ

**User Story:** ในฐานะผู้ใช้ ฉันต้องการลงทะเบียนและเข้าสู่ระบบ เพื่อให้ข้อมูลทางการเงินของฉันถูกเก็บอย่างปลอดภัยและเป็นส่วนตัว

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. WHEN User กรอกอีเมลและรหัสผ่านที่ถูกต้องตามรูปแบบ, THE System SHALL สร้างบัญชีผู้ใช้ใหม่และส่งอีเมลยืนยัน
2. WHEN User กรอกอีเมลและรหัสผ่านเพื่อเข้าสู่ระบบ, THE System SHALL ตรวจสอบข้อมูลและอนุญาตให้เข้าถึงระบบภายใน 2 วินาที
3. IF User กรอกรหัสผ่านผิดเกิน 5 ครั้งติดต่อกัน, THEN THE System SHALL ล็อกบัญชีชั่วคราวเป็นเวลา 15 นาที
4. IF User ใช้อีเมลที่ลงทะเบียนแล้ว, THEN THE System SHALL แสดงข้อความแจ้งว่าอีเมลนี้ถูกใช้งานแล้ว

---

### ข้อกำหนดที่ 2: การบันทึกรายรับ

**User Story:** ในฐานะผู้ใช้ ฉันต้องการบันทึกรายรับ เพื่อให้สามารถติดตามแหล่งที่มาของรายได้ทั้งหมด

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. WHEN User กรอกข้อมูลรายรับ (จำนวนเงิน, หมวดหมู่, วันที่, หมายเหตุ), THE System SHALL บันทึก Transaction ประเภทรายรับและแสดงการยืนยัน
2. THE System SHALL กำหนดให้ฟิลด์จำนวนเงินและวันที่เป็นข้อมูลบังคับสำหรับ Transaction ทุกรายการ
3. WHEN User ไม่ระบุหมวดหมู่, THE System SHALL กำหนดหมวดหมู่เป็น "ไม่ระบุ" โดยอัตโนมัติ
4. IF User กรอกจำนวนเงินเป็นค่าลบหรือศูนย์, THEN THE System SHALL แสดงข้อความแจ้งเตือนว่าจำนวนเงินต้องมากกว่าศูนย์

---

### ข้อกำหนดที่ 3: การบันทึกรายจ่าย

**User Story:** ในฐานะผู้ใช้ ฉันต้องการบันทึกรายจ่าย เพื่อให้สามารถติดตามการใช้จ่ายทั้งหมด

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. WHEN User กรอกข้อมูลรายจ่าย (จำนวนเงิน, หมวดหมู่, วันที่, หมายเหตุ), THE System SHALL บันทึก Transaction ประเภทรายจ่ายและแสดงการยืนยัน
2. THE System SHALL กำหนดให้ฟิลด์จำนวนเงินและวันที่เป็นข้อมูลบังคับสำหรับ Transaction ทุกรายการ
3. WHEN User ไม่ระบุหมวดหมู่, THE System SHALL กำหนดหมวดหมู่เป็น "ไม่ระบุ" โดยอัตโนมัติ
4. IF User กรอกจำนวนเงินเป็นค่าลบหรือศูนย์, THEN THE System SHALL แสดงข้อความแจ้งเตือนว่าจำนวนเงินต้องมากกว่าศูนย์

---

### ข้อกำหนดที่ 4: การจัดการหมวดหมู่

**User Story:** ในฐานะผู้ใช้ ฉันต้องการสร้างและจัดการหมวดหมู่ เพื่อให้สามารถจัดกลุ่มรายรับรายจ่ายตามประเภทที่ต้องการ

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. THE System SHALL จัดเตรียมหมวดหมู่เริ่มต้น (อาหาร, ค่าเดินทาง, ที่พัก, เงินเดือน, รายได้เสริม, อื่นๆ) ให้ User ใหม่ทุกคน
2. WHEN User สร้าง Category ใหม่พร้อมระบุชื่อและไอคอน, THE System SHALL บันทึก Category และแสดงในรายการหมวดหมู่
3. WHEN User แก้ไขชื่อ Category, THE System SHALL อัปเดตชื่อ Category ในทุก Transaction ที่เกี่ยวข้อง
4. IF User ลบ Category ที่มี Transaction อ้างอิงอยู่, THEN THE System SHALL ย้าย Transaction เหล่านั้นไปยังหมวดหมู่ "ไม่ระบุ" ก่อนลบ

---

### ข้อกำหนดที่ 5: การแก้ไขและลบรายการ

**User Story:** ในฐานะผู้ใช้ ฉันต้องการแก้ไขและลบรายการที่บันทึกไว้ เพื่อให้ข้อมูลทางการเงินถูกต้องเสมอ

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. WHEN User เลือกแก้ไข Transaction, THE System SHALL แสดงฟอร์มพร้อมข้อมูลเดิมและอนุญาตให้แก้ไขทุกฟิลด์
2. WHEN User ยืนยันการแก้ไข Transaction, THE System SHALL บันทึกการเปลี่ยนแปลงและอัปเดตข้อมูลสรุปทันที
3. WHEN User เลือกลบ Transaction, THE System SHALL แสดงข้อความยืนยันก่อนดำเนินการลบ
4. WHEN User ยืนยันการลบ Transaction, THE System SHALL ลบรายการและอัปเดตข้อมูลสรุปทันที

---

### ข้อกำหนดที่ 6: แดชบอร์ดภาพรวม

**User Story:** ในฐานะผู้ใช้ ฉันต้องการเห็นภาพรวมทางการเงินในหน้าเดียว เพื่อให้เข้าใจสถานะการเงินปัจจุบันอย่างรวดเร็ว

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. WHEN User เข้าสู่ Dashboard, THE System SHALL แสดงยอดรวมรายรับ ยอดรวมรายจ่าย และยอดคงเหลือของเดือนปัจจุบัน
2. THE System SHALL แสดงรายการ Transaction ล่าสุด 10 รายการบน Dashboard
3. WHEN User เปลี่ยนช่วงเวลาบน Dashboard, THE System SHALL อัปเดตข้อมูลสรุปตามช่วงเวลาที่เลือกภายใน 1 วินาที
4. THE Dashboard SHALL แสดงกราฟแท่งเปรียบเทียบรายรับและรายจ่ายรายเดือน

---

### ข้อกำหนดที่ 7: การวิเคราะห์ตามหมวดหมู่

**User Story:** ในฐานะผู้ใช้ ฉันต้องการวิเคราะห์รายจ่ายตามหมวดหมู่ เพื่อให้ทราบว่าใช้จ่ายในด้านใดมากที่สุด

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. WHEN User เลือกดูการวิเคราะห์ตามหมวดหมู่, THE Analytics_Module SHALL แสดงกราฟวงกลม (pie chart) แสดงสัดส่วนรายจ่ายแต่ละ Category
2. THE Analytics_Module SHALL แสดงตารางจัดอันดับ Category ที่มีรายจ่ายสูงสุดถึงต่ำสุด
3. WHEN User เลือก Category ในกราฟ, THE Analytics_Module SHALL แสดงรายละเอียด Transaction ทั้งหมดใน Category นั้น
4. THE Analytics_Module SHALL คำนวณเปอร์เซ็นต์การเปลี่ยนแปลงของแต่ละ Category เทียบกับเดือนก่อนหน้า

---

### ข้อกำหนดที่ 8: การวิเคราะห์แนวโน้มตามช่วงเวลา

**User Story:** ในฐานะผู้ใช้ ฉันต้องการเห็นแนวโน้มรายรับรายจ่ายตามช่วงเวลา เพื่อให้สามารถวางแผนการเงินในอนาคต

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. WHEN User เลือกดูแนวโน้มรายเดือน, THE Analytics_Module SHALL แสดงกราฟเส้น (line chart) แสดงรายรับและรายจ่ายย้อนหลัง 12 เดือน
2. WHEN User เลือกดูแนวโน้มรายสัปดาห์, THE Analytics_Module SHALL แสดงกราฟเส้นแสดงรายรับและรายจ่ายย้อนหลัง 12 สัปดาห์
3. THE Analytics_Module SHALL แสดงค่าเฉลี่ยรายรับและรายจ่ายต่อเดือนในช่วงเวลาที่เลือก
4. THE Analytics_Module SHALL แสดงเดือนที่มีรายจ่ายสูงสุดและต่ำสุดในช่วงเวลาที่เลือก

---

### ข้อกำหนดที่ 9: การวิเคราะห์เปรียบเทียบ

**User Story:** ในฐานะผู้ใช้ ฉันต้องการเปรียบเทียบข้อมูลทางการเงินระหว่างช่วงเวลาต่างๆ เพื่อให้เห็นความแตกต่างและปรับปรุงพฤติกรรมการใช้จ่าย

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. WHEN User เลือกเปรียบเทียบ 2 เดือน, THE Analytics_Module SHALL แสดงกราฟแท่งเปรียบเทียบรายรับรายจ่ายของทั้ง 2 เดือนแบบเคียงข้างกัน
2. THE Analytics_Module SHALL คำนวณและแสดงเปอร์เซ็นต์การเปลี่ยนแปลงระหว่างช่วงเวลาที่เปรียบเทียบ
3. WHEN User เลือกเปรียบเทียบตาม Category, THE Analytics_Module SHALL แสดงการเปลี่ยนแปลงของแต่ละ Category ระหว่างช่วงเวลาที่เลือก

---

### ข้อกำหนดที่ 10: การนำเข้าและส่งออกข้อมูล

**User Story:** ในฐานะผู้ใช้ ฉันต้องการนำเข้าและส่งออกข้อมูลทางการเงิน เพื่อให้สามารถสำรองข้อมูลหรือย้ายข้อมูลจากระบบอื่น

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. WHEN User อัปโหลดไฟล์ CSV ที่มีรูปแบบถูกต้อง, THE Parser SHALL แปลงข้อมูลเป็น Transaction และบันทึกเข้าระบบ
2. IF User อัปโหลดไฟล์ CSV ที่มีรูปแบบไม่ถูกต้อง, THEN THE Parser SHALL แสดงรายละเอียดข้อผิดพลาดพร้อมระบุหมายเลขแถวที่มีปัญหา
3. WHEN User เลือกส่งออกข้อมูล, THE Pretty_Printer SHALL จัดรูปแบบ Transaction ทั้งหมดเป็นไฟล์ CSV ที่ถูกต้อง
4. FOR ALL Transaction ที่ส่งออกแล้วนำเข้ากลับ, THE System SHALL สร้าง Transaction ที่มีข้อมูลเทียบเท่ากับต้นฉบับ (round-trip property)

---

### ข้อกำหนดที่ 11: การกรองและค้นหารายการ

**User Story:** ในฐานะผู้ใช้ ฉันต้องการกรองและค้นหารายการทางการเงิน เพื่อให้สามารถค้นหารายการที่ต้องการได้อย่างรวดเร็ว

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. WHEN User กรอง Transaction ตาม Category, THE System SHALL แสดงเฉพาะ Transaction ที่อยู่ใน Category ที่เลือก
2. WHEN User กรอง Transaction ตามช่วงวันที่, THE System SHALL แสดงเฉพาะ Transaction ที่อยู่ในช่วงวันที่ที่ระบุ
3. WHEN User กรอง Transaction ตามประเภท (รายรับ/รายจ่าย), THE System SHALL แสดงเฉพาะ Transaction ประเภทที่เลือก
4. WHEN User ค้นหาด้วยคำค้น, THE System SHALL แสดง Transaction ที่มีหมายเหตุตรงกับคำค้นภายใน 500 มิลลิวินาที

---

### ข้อกำหนดที่ 12: การออกแบบที่ตอบสนอง (Responsive Design)

**User Story:** ในฐานะผู้ใช้ ฉันต้องการใช้งานระบบได้ทั้งบนคอมพิวเตอร์และมือถือ เพื่อให้สามารถบันทึกรายการได้ทุกที่ทุกเวลา

#### เกณฑ์การยอมรับ (Acceptance Criteria)

1. THE System SHALL แสดงผลได้อย่างถูกต้องบนหน้าจอที่มีความกว้างตั้งแต่ 320px ถึง 1920px
2. WHILE User ใช้งานบนอุปกรณ์มือถือ (หน้าจอกว้างน้อยกว่า 768px), THE System SHALL แสดงเมนูนำทางแบบ hamburger menu
3. THE System SHALL รองรับการใช้งานผ่าน touch gesture บนอุปกรณ์มือถือ
4. WHILE User ใช้งานบนอุปกรณ์มือถือ, THE System SHALL แสดงกราฟในรูปแบบที่อ่านได้ง่ายโดยรองรับการซูมและเลื่อน
